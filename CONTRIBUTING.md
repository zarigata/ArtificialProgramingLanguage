## Contribution Process

### 1. Discussion First

Before making significant contributions:
1. Open an issue describing your proposal
2. Discuss with maintainers and community
3. Reach consensus on approach
4. Get approval to proceed

### 2. Design Documents

For major features:
1. Create a design document
2. Include rationale, alternatives, and trade-offs
3. Submit for review
4. Iterate based on feedback
5. Get approval before implementation

## Design Principles

All contributions should align with core principles:

### 1. AI-First Design
- Optimize for AI comprehension and generation
- Predictable, learnable patterns
- Minimal ambiguity
- Rich type information

### 2. Performance
- Zero-cost abstractions
- Compile-time optimization
- Direct hardware access
- Predictable performance

### 3. Safety
- Memory safety without GC
- Thread safety
- Type safety
- Compile-time guarantees

### 4. Simplicity
- Minimal language surface
- Orthogonal features
- Clear semantics
- Avoid special cases

### 5. Interoperability
- C FFI
- Python bindings
- WASM target
- Existing ecosystem integration

## Proposal Template

### Feature Proposal

```markdown
# Feature Name

## Summary
Brief description of the feature.

## Motivation
Why is this feature needed? What problems does it solve?

## Design
Detailed design of the feature.

### Syntax
How will users write this feature?

### Semantics
What does it mean? How does it behave?

### Implementation
How will it be implemented?

## Alternatives
What other approaches were considered?

## Trade-offs
What are the pros and cons?

## AI Considerations
How does this affect AI code generation?

## Examples
Show concrete usage examples.

## Open Questions
What needs to be resolved?
```

## Style Guide

### Language Design

**Naming Conventions**:
- `snake_case` for variables, functions, modules
- `PascalCase` for types, traits
- `SCREAMING_SNAKE_CASE` for constants

**Syntax**:
- Consistent keyword placement
- Clear delimiters
- Minimal punctuation
- Explicit over implicit

### Documentation

**Format**: Markdown
**Structure**:
- Clear headings
- Code examples
- Explanations
- Edge cases

**Tone**:
- Clear and concise
- Beginner-friendly
- Technically accurate
- No jargon without explanation
